What is Vue.js?

Vue.js is a progressive JavaScript framework for building user interfaces. You can use it as a lightweight view layer in a single HTML page, or scale it into a full single-page application with routing, state management, testing, and server rendering.

Vue focuses on clarity, reactive state, and incremental adoption. It gives you declarative templates, a powerful reactivity system, single-file components, and a mature ecosystem without forcing a heavy architectural style on day one.

Progressive

Start with a CDN script in a static page or adopt the full tooling stack with Vite, Router, and Pinia.

Reactive

Data changes update the DOM automatically through refs, reactive objects, computed values, and watchers.

Single-File Components

Keep template, logic, and scoped styles in one .vue file for maintainable components and better team readability.

Strong Ecosystem

Vue Router, Pinia, DevTools, Nuxt, Vue Test Utils, and Vite cover the normal needs of production teams.

Why Vue?

Vue is often chosen because it balances developer experience with production capability. The learning curve is gentle for beginners, but the architecture scales well when teams need routing, stores, testing, lazy loading, and SSR.

Fast to Learn

Templates feel close to plain HTML, directives are readable, and Vue hides a lot of boilerplate without hiding the fundamentals.

Flexible API Style

You can read old code written in the Options API and build modern code with Composition API and script setup.

Production Friendly

Code splitting, transitions, SSR via Nuxt, typed code, and store patterns are well documented and stable.

Vue Learning Path

A strong Vue roadmap is not random. Learn the features in the order they actually depend on each other.

1Templates First

Interpolation, v-bind, v-on, v-if, v-show, v-for, and v-model.

2Understand Reactivity

Know when to use ref(), reactive(), computed(), watch(), and watchEffect().

3Component Communication

Master props, emits, slots, provide/inject, and custom v-model before reaching for global stores.

4Composition API

Extract reusable logic into composables, learn <script setup>, and type your code cleanly.

5Ecosystem & Production

Vue Router, Pinia, API patterns, testing, performance, SSR, environment handling, and deployment.

6Practice Projects

Build forms, dashboards, CRUD apps, and route-based products so the concepts become muscle memory.

Pragmatic rule: learn enough Options API to read existing codebases, but build new code mostly with Composition API and <script setup>. That is the most common modern Vue style.

Reactivity Fundamentals

Vue tracks dependencies automatically. When you read a reactive value during render or in a computed/watcher, Vue knows what should update later.

ref()

Best for primitives and single values like booleans, strings, numbers, DOM refs, and async state flags.

reactive()

Best for objects that you want to mutate deeply without constantly accessing .value.

computed()

Derived state. Use for filtered lists, totals, formatted output, and reusable logic in templates.

shallowRef()

Useful when you want Vue to track only the top-level reference, such as large third-party instances or immutable payloads.

import { ref, reactive, toRefs, nextTick } from 'vue';

const count = ref(0);
const filters = reactive({
  search: '',
  inStock: false,
});
const inputRef = ref(null);

async function incrementAndFocus() {
  count.value += 1;
  await nextTick();
  inputRef.value?.focus();
}

const state = reactive({
  page: 1,
  pageSize: 10,
});

const { page, pageSize } = toRefs(state);

Common trap: do not destructure a reactive() object directly if you still need reactivity. Use toRefs() or keep property access on the original object.

import { ref, computed, watch, watchEffect } from 'vue';

const firstName = ref('Ada');
const lastName = ref('Lovelace');

const results = ref([]);

watch(query, async (term, _oldTerm, onCleanup) => {
  const controller = new AbortController();
  onCleanup(() => controller.abort());

  if (!term) {
    results.value = [];
    return;
  }

  const response = await fetch(`/api/search?q=${encodeURIComponent(term)}`, {
    signal: controller.signal,
  });

  results.value = await response.json();
}, { immediate: true, flush: 'post' });

});

Form Validation

Good Vue forms validate at three levels: input-level feedback, form-level submit rules, and async server validation. Do not wait until submit to tell users everything is wrong.

Synchronous rules

Required fields, min length, email shape, number ranges, and cross-field checks like password confirmation.

Async checks

Username uniqueness, coupon validity, and server-side business rules should return field-specific feedback.

Touched state

Show errors after blur or submit so the form feels guided, not hostile from the first keypress.

<script setup>
import { computed, reactive, ref, watch } from 'vue';

const form = reactive({
  name: '',
  email: '',
  password: '',
  confirmPassword: '',
});

const touched = reactive({
  name: false,
  email: false,
  password: false,
  confirmPassword: false,
});

const serverError = ref('');
const checkingEmail = ref(false);

const errors = computed(() => ({
  name: form.name.trim().length >= 3 ? '' : 'Name must be at least 3 characters.',
  email: /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(form.email) ? '' : 'Enter a valid email address.',
  password: form.password.length >= 8 ? '' : 'Password must be at least 8 characters.',
  confirmPassword: form.password === form.confirmPassword ? '' : 'Passwords must match.',
}));

watch(() => form.email, async (email, _oldEmail, onCleanup) => {
  if (!email || errors.value.email) return;

  const controller = new AbortController();
  onCleanup(() => controller.abort());
  checkingEmail.value = true;

  try {
    const response = await fetch(`/api/check-email?email=${encodeURIComponent(email)}`, {
      signal: controller.signal,
    });
    const result = await response.json();
    serverError.value = result.available ? '' : 'This email is already registered.';
  } catch (error) {
    if (error.name !== 'AbortError') {
      serverError.value = 'Could not validate email right now.';
    }
  } finally {
    checkingEmail.value = false;
  }
});

const canSubmit = computed(() =>
  Object.values(errors.value).every(value => !value) && !serverError.value
);

function handleBlur(field) {
  touched[field] = true;
}

function submitForm() {
  Object.keys(touched).forEach(key => {
    touched[key] = true;
  });
  if (!canSubmit.value) return;
  console.log('Submit payload', form);
}
</script>

<template>
  <form @submit.prevent="submitForm" class="d-grid gap-3" novalidate>
    <div>
      <input v-model.trim="form.name" @blur="handleBlur('name')" placeholder="Name" />
      <p v-if="touched.name && errors.name" class="text-danger small mb-0">{{ errors.name }}</p>
    </div>

    <div>
      <input v-model.trim="form.email" @blur="handleBlur('email')" type="email" placeholder="Email" />
      <p v-if="checkingEmail" class="small mb-0">Checking email...</p>
      <p v-else-if="(touched.email && errors.email) || serverError" class="text-danger small mb-0">
        {{ errors.email || serverError }}
      </p>
    </div>

    <div>
      <input v-model="form.password" @blur="handleBlur('password')" type="password" placeholder="Password" />
      <p v-if="touched.password && errors.password" class="text-danger small mb-0">{{ errors.password }}</p>
    </div>

    <div>
      <input v-model="form.confirmPassword" @blur="handleBlur('confirmPassword')" type="password" placeholder="Confirm password" />
      <p v-if="touched.confirmPassword && errors.confirmPassword" class="text-danger small mb-0">{{ errors.confirmPassword }}</p>
    </div>

    <button :disabled="!canSubmit">Create account</button>
  </form>
</template>

Composition API

Composition API groups logic by feature rather than by option type. That becomes much more maintainable as components grow.

Good for large components

Related state, computed values, watchers, and methods can stay together instead of being split across data, methods, and computed.

Great for reuse

Move logic into composables such as useCart, usePermissions, or useInfiniteScroll.

// composables/useCounter.js
import { ref, computed } from 'vue';

export function useCounter(initialValue = 0) {
  const count = ref(initialValue);

  const doubled = computed(() => count.value * 2);

  function increment() {
    count.value += 1;
  }

  function reset() {
    count.value = initialValue;
  }

  return {
    count,
    doubled,
    increment,
    reset,
  };
}

Pinia State Management

Pinia is the official state management library for Vue. It is lightweight, typed well, and much easier to reason about than older Vuex patterns.

// stores/cart.ts
import { computed, ref } from 'vue';
import { defineStore } from 'pinia';

export const useCartStore = defineStore('cart', () => {
  const items = ref<Array<{ id: number; title: string; price: number; qty: number }>>([]);

  const total = computed(() =>
    items.value.reduce((sum, item) => sum + item.price * item.qty, 0)
  );

  function addItem(product: { id: number; title: string; price: number }) {
    const existing = items.value.find(item => item.id === product.id);
    if (existing) {
      existing.qty += 1;
      return;
    }
    items.value.push({ ...product, qty: 1 });
  }

  function clearCart() {
    items.value = [];
  }

  return { items, total, addItem, clearCart };
});

API Integration

Fetching data in Vue is usually done inside a composable or lifecycle hook. The main production rules are simple: track loading, surface errors, and cancel stale requests.

import { ref, watchEffect } from 'vue';

export function useFetchJson(urlRef) {
  const data = ref(null);
  const loading = ref(false);
  const error = ref('');

  watchEffect(async (onCleanup) => {
    const url = urlRef.value;
    if (!url) return;

    const controller = new AbortController();
    onCleanup(() => controller.abort());

    loading.value = true;
    error.value = '';

    try {
      const response = await fetch(url, { signal: controller.signal });
      if (!response.ok) throw new Error('Request failed');
      data.value = await response.json();
    } catch (err) {
      if (err.name !== 'AbortError') {
        error.value = err.message;
      }
    } finally {
      loading.value = false;
    }
  });

  return { data, loading, error };
}

Architecture advice: keep raw fetch logic in composables or service modules. Components should mainly coordinate state and render UI.

  email: /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(form.email) ? '' : 'Enter a valid email.',

  Object.keys(serverErrors).forEach(key => delete serverErrors[key]);
}

async function submitForm() {
  successMessage.value = '';
  resetServerErrors();

  if (Object.values(clientErrors.value).some(Boolean)) {
    Object.assign(serverErrors, clientErrors.value);
    return;

    form.email = '';
    form.message = '';

3Pinia Store Flow
import { defineStore } from 'pinia';
import { ref } from 'vue';

type User = {
  id: number;
  name: string;
  email: string;
  status: 'active' | 'blocked';
};

export const useUsersStore = defineStore('users', () => {
  const items = ref<User[]>([]);
  const loading = ref(false);
  const error = ref('');

  async function fetchUsers() {
    loading.value = true;
    error.value = '';
    try {
      const response = await fetch('/api/users');
      items.value = await response.json();
    } catch (err) {
      error.value = 'Could not load users.';
    } finally {
      loading.value = false;
    }
  }

  async function createUser(payload: Omit<User, 'id'>) {
    const tempUser = { id: Date.now(), ...payload };
    items.value.unshift(tempUser);

    try {
      const response = await fetch('/api/users', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(payload),
      });
      const createdUser = await response.json();
      const index = items.value.findIndex(user => user.id === tempUser.id);
      if (index >= 0) items.value[index] = createdUser;
    } catch (err) {
      items.value = items.value.filter(user => user.id !== tempUser.id);
      throw err;
    }
  }

  async function updateUser(id: number, payload: Partial<User>) {
    const existing = items.value.find(user => user.id === id);
    if (!existing) return;

    const snapshot = { ...existing };
    Object.assign(existing, payload);

    try {
      await fetch(`/api/users/${id}`, {
        method: 'PATCH',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(payload),
      });
    } catch (err) {
      Object.assign(existing, snapshot);
      throw err;
    }
  }

  async function removeUser(id: number) {
    const snapshot = [...items.value];
    items.value = items.value.filter(user => user.id !== id);

    try {
      await fetch(`/api/users/${id}`, { method: 'DELETE' });
    } catch (err) {
      items.value = snapshot;
      throw err;
    }
  }

  return { items, loading, error, fetchUsers, createUser, updateUser, removeUser };
});

Interview Questions and Pitfalls

QuestionShort answer
What is the difference between ref() and reactive()?ref() wraps single values and uses .value; reactive() wraps objects and arrays with deep proxy tracking.
When do you use computed() instead of watch()?Use computed() for derived state and watch() for side effects like fetches, local storage, or analytics.
Why are keys important in v-for?They help Vue reuse or replace the correct DOM nodes and component instances during updates.
When should you choose Pinia over props and emits?When multiple distant parts of the app depend on the same state or behavior and local component communication becomes awkward.
What does <KeepAlive> do?It caches inactive component instances so their local state survives tab or route switches.
What is the purpose of defineModel()?It simplifies custom v-model support in reusable components.

Common mistakes to avoid:

  • Using watch() where a computed value would be simpler.
  • Mutating props directly instead of emitting updates.
  • Using array indexes as v-for keys in dynamic lists.
  • Fetching directly in many components instead of centralizing logic in composables.
  • Putting server secrets into client-exposed VITE_* variables.
