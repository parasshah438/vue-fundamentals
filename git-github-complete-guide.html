What Is Git and What Is GitHub?

Git is a distributed version control system that tracks file changes locally. GitHub is a cloud platform built around Git repositories, collaboration, pull requests, issues, releases, automation, and code hosting.

Git = Version Control Engine

Git records snapshots of your files so you can inspect history, branch safely, compare changes, and recover from mistakes.

Runs on your machine
GitHub = Collaboration Platform

GitHub stores Git repositories remotely and adds pull requests, review, issues, releases, Actions, Pages, Discussions, Projects, and security scans.

Runs in the cloud
Topic Git GitHub
PurposeTrack code history and file changesHost repositories and coordinate teams
Works offline?Yes, almost everythingNo, needs internet for hosted features
Key conceptsCommit, branch, merge, rebase, stashRepository, PR, issue, release, action
Who uses it?Every developer touching source controlTeams, open-source maintainers, DevOps pipelines
Can you use one without the other?YesGitHub repositories still use Git underneath

Why Learn Git from Basic to Advanced?

Git starts as a save-history tool, but at professional level it becomes a release system, audit trail, collaboration layer, recovery tool, and automation trigger.

Time Machine

Inspect who changed what, when, and why.

Safe Experimentation

Branches let you try risky ideas without breaking main.

Team Collaboration

PRs, reviews, approvals, and branch protection enforce quality.

Reliable Releases

Tags, release branches, and CI pipelines make shipping repeatable.

Recovery

Reflog, revert, restore, stash, and bisect help recover from mistakes fast.

Career Skill

Git questions show up in interviews, code tests, and production incident work.

Bottom line: if you can branch cleanly, review responsibly, recover safely, and automate releases, you are operating at team level rather than tutorial level.

Daily Git Workflow

Most professional Git work follows the same loop: update your branch, make a focused change, review the diff, commit with a meaningful message, push, and open or update a pull request.

1
Sync

Pull the latest remote changes before starting new work.

2
Edit

Make a focused change on a feature or fix branch.

3
Review

Check git diff and stage only the intended files.

4
Share

Commit, push, open a PR, then respond to review.

git switch main
git pull --rebase origin main

git switch -c feature/profile-card
# edit files
git status
git diff
git add src/profile.js src/profile.css
git commit -m "feat: add reusable profile card"
git push -u origin feature/profile-card

Good habit: run git diff --staged before every commit. It catches accidental secrets, debug statements, and unrelated files before they leave your machine.

Status, Add, Commit, and the Basic Snapshot Cycle

The core Git loop is simple: check status, stage the right files, review what is staged, commit with a useful message, then inspect history.

git status
git add src/app.js
git add src/styles.css
git diff --staged
git commit -m "feat: add pricing card layout"
git log --oneline --decorate -5
CommandWhat it doesShort example
git statusShows modified, staged, and untracked files.git status
git add fileStages a specific file for the next commit.git add app.js
git add .Stages all changes under the current directory.git add .
git diffShows unstaged changes.git diff
git diff --stagedShows staged changes that will be committed.git diff --staged
git commit -m "message"Creates a new snapshot with a message.git commit -m "fix: navbar overlap"

Commit message rule: describe the intent, not the keystrokes. "fix: prevent duplicate order submit" is useful later. "update code" is not.

Branching from Basic to Team-Level

Branches let you isolate features, fixes, experiments, or release work. Modern Git uses git switch for branch movement because it is clearer than overloading git checkout.

git branch
git branch -a
git switch -c feature/auth-form
git push -u origin feature/auth-form
git switch main
git pull --rebase origin main
git switch feature/auth-form
git merge main
main

Production-ready branch protected by review and CI.

feature/*

Short-lived branches for individual tasks or PRs.

hotfix/*

Urgent fixes made from the current release line.

Merge, Rebase, and Cherry-Pick

All three combine history, but they solve different problems. Knowing when to use each one is a common interview and production question.

ToolBest useMain tradeoff
mergePreserve exact branch historyHistory graph can become noisy
rebaseKeep feature branch history linear before mergeRewrites commits, so avoid on shared public branches
cherry-pickCopy one specific commit to another branchCan duplicate logic if overused
1Merge
git switch main
git merge feature/cart-discount
2Rebase
git switch feature/cart-discount
git fetch origin
git rebase origin/main
3Cherry-Pick
git switch release/1.2
git cherry-pick a1b2c3d

Safe rule: rebasing your own local feature branch is fine. Rebasing a branch other people are already using usually creates unnecessary pain.

Interactive Rebase and Conflict Resolution

Interactive rebase is the best way to polish your own local feature branch before review. It is also where many developers first hit serious conflicts, so the recovery flow should feel routine, not scary.

1Clean Up Local History
git fetch origin
git switch feature/cart-discount
git rebase -i origin/main

pick a1b2c3d add cart service
fixup b2c3d4e fix typo in cart service
reword c3d4e5f rename discount helper

Use pick, reword, squash, fixup, or drop to turn noisy commits into a reviewable story.

2Use Autosquash for Follow-Up Fixes
git commit --fixup a1b2c3d
git commit --fixup c3d4e5f
git rebase -i --autosquash origin/main

Autosquash saves time when you want tiny correction commits folded into earlier work without hand-editing the whole todo list.

3Resolve Conflicts Without Panic
git status
# open conflicted files and fix them
git add src/cart.js
git rebase --continue

# if you want to stop
git rebase --abort
git merge --abort
<<<<<<< HEAD
const taxRate = 0.18;
=======
const taxRate = 0.20;
>>>>>>> feature/cart-discount

Keep the final intended code, delete the conflict markers, stage the file, and continue the rebase or merge.

Safe team rule: interactive rebase is ideal for your own unpublished branch. Once others are building on that branch, coordinate before rewriting history.

Remotes, Fetch, Pull, Push, and Upstream

Git remotes connect your local repository to hosted copies like GitHub. Most confusion comes from not distinguishing fetch, pull, and push.

origin

Your default remote repository.

fetch

Downloads remote history but does not merge it.

pull

Fetches plus integrates remote changes.

push

Uploads your commits to the remote branch.

git remote -v
git fetch origin
git pull --rebase origin main
git push origin main
git push -u origin feature/profile-card
git remote add upstream https://github.com/company/project.git

Why many teams prefer git pull --rebase: it keeps your local branch from collecting extra merge commits every time you sync with main.

History, Diff, Show, Blame, and Search

Reading history well is as important as writing history well. These commands help you review changes, audit a bug, and understand why a file looks the way it does.

git log --oneline --graph --decorate --all
git show a1b2c3d
git diff main..feature/search
git blame src/auth.js
git grep "TODO"
git shortlog -sn
CommandShort descriptionShort example
git logBrowse commit history.git log --stat -5
git show shaInspect one commit in detail.git show a1b2c3d
git diffCompare changes between states.git diff HEAD~1
git blame fileShow who last changed each line.git blame api.php

Fork Workflow and Upstream Sync

Fork-based collaboration is standard in open source and useful anywhere contributors should not push directly to the main repository. The key idea is simple: origin is your fork, upstream is the original project.

1Fork and Clone
gh repo fork owner/project --clone
cd project
git remote -v
2Add Upstream and Sync
git remote add upstream https://github.com/owner/project.git
git fetch upstream
git switch main
git merge --ff-only upstream/main
git push origin main
3Open a Contribution PR
git switch -c fix/readme-typo
# make changes
git push -u origin fix/readme-typo
gh pr create --repo owner/project --fill
RemoteUsually meansTypical action
originYour fork under your accountPush branches and keep your fork updated
upstreamThe original project you want to contribute toFetch latest changes and open PRs against it

Practical rule: keep your fork's main branch clean and branch from it for each contribution. Do not stack unrelated features into one PR branch.

GitHub Authentication: HTTPS, Personal Access Tokens, and SSH

GitHub no longer accepts account passwords for Git over HTTPS. Use either a Personal Access Token or SSH keys.

1HTTPS + Personal Access Token
  • Create a token in GitHub settings.
  • Use the token when Git prompts for a password.
  • Store it in a credential manager, not a text file.
2SSH Key Setup
ssh-keygen -t ed25519 -C "you@example.com"
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
cat ~/.ssh/id_ed25519.pub

Copy the public key into GitHub under Settings > SSH and GPG keys.

ssh -T git@github.com
git remote set-url origin git@github.com:your-name/repo-name.git

Practical default: if this is your main development machine, SSH is usually the cleanest long-term option.

Signed Commits, Signed Tags, and Verified History

Signing proves that a commit or tag was created with your trusted key. On GitHub, correctly configured signatures show a Verified badge and help teams trust release and audit history.

1Enable SSH-Based Signing
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519.pub
git config --global commit.gpgsign true
git config --global tag.gpgSign true

This reuses an SSH public key you already trust for GitHub auth and makes signing the default for future commits and tags.

2Create and Verify Signed History
git commit -S -m "feat: add checkout summary"
git tag -s v1.4.0 -m "Release 1.4.0"
git log --show-signature -1
git tag -v v1.4.0

Signed tags matter most for releases because they give you a verifiable marker for what actually shipped.

TypeBest useShort note
SSH signingTeams already using SSH keys with GitHubSimpler setup for many developers
GPG signingOlder workflows or orgs standardized on GPGStill valid, but usually more setup
Signed commitsVerify author identity per commitUseful for protected branches and audits
Signed tagsVerify releasesHigher value than lightweight tags for shipping history

Important: GitHub must know the same public key before it can show commits or tags as Verified, so add the signing key in Settings > SSH and GPG keys.

GitHub Actions CI/CD

GitHub Actions lets you automatically run tests, builds, linting, deploys, and release tasks whenever code changes land.

name: Node CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm test -- --runInBand
  • Store secrets in repository or environment secrets, never in workflow files.
  • Use required status checks on protected branches.
  • Keep workflows fast enough that developers do not bypass them.

Deploy Static Projects with GitHub Pages

GitHub Pages is ideal for documentation, portfolio sites, static demos, and front-end experiments.

name: Deploy Pages

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/configure-pages@v5
      - uses: actions/upload-pages-artifact@v3
        with:
          path: .

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4

Best fit: GitHub Pages works for static output. If you need a full backend, use another host and keep GitHub as the source and CI provider.

Undo, Restore, Amend, Reset, and Revert

Undo commands are where many developers get nervous. The safe approach is to distinguish whether you want to change the working tree, the index, or commit history.

git restore app.js
git restore --staged app.js
git commit --amend -m "fix: correct login validation message"
git reset --soft HEAD~1
git reset --mixed HEAD~1
git revert a1b2c3d
CommandUse it when
git restore fileYou want to throw away working-directory changes in one file.
git restore --staged fileYou staged a file by mistake and want to unstage it.
git commit --amendYou want to fix the last commit before sharing it.
git reset --soft HEAD~1You want to undo the last commit but keep changes staged.
git revert shaYou need to safely undo a commit that may already be shared.

Production-safe rule: prefer revert over history rewriting when the commit is already on a shared branch.

Stash, Clean, and Temporary Work

Stash is useful when you need to switch context without committing half-done work. Clean removes untracked files. Both are powerful, so use them deliberately.

git stash push -m "wip: checkout form redesign"
git stash list
git stash show -p stash@{0}
git stash pop
git clean -fd
git clean -fdX
  • git stash push -m "message" is better than anonymous stash entries.
  • git clean -fd removes untracked files and directories.
  • git clean -fdX also removes ignored files like build output. Use carefully.

Reflog, Detached HEAD, and Recovery Tricks

Reflog is Git's local record of where your HEAD and branch references recently pointed. It is one of the most useful rescue tools in Git.

git reflog
git reset --hard HEAD@{1}
git branch recover-work a1b2c3d
git fsck --lost-found
Detached HEAD

You checked out a commit directly instead of a branch. Create a branch if you want to keep new work.

Reflog Scope

Reflog is local history. It helps recover locally lost pointers, but it is not a remote backup.

.gitattributes, Line Endings, and Cross-Platform Safety

.gitignore decides what should stay out of the repository. .gitattributes decides how tracked files behave once they are in it. This matters most on mixed Windows, macOS, and Linux teams.

1Recommended .gitattributes File
* text=auto
*.sh text eol=lf
*.bat text eol=crlf
*.png binary
*.jpg binary
*.psd filter=lfs diff=lfs merge=lfs -text

This keeps text files normalized, forces the right shell-script endings, and marks binaries so Git does not try to diff them like text.

2Inspect and Renormalize Once
git ls-files --eol
git add --renormalize .
git diff --cached --check
git commit -m "chore: normalize line endings"

Run the renormalize commit once after introducing .gitattributes, then let the file enforce policy going forward.

SituationRecommended approachWhy
Cross-platform team repoCommit a real .gitattributes fileRepository policy is safer than relying on each developer's machine config
Windows machine without repo policygit config --global core.autocrlf trueReduces accidental LF/CRLF churn in simple repos
macOS or Linux machinegit config --global core.autocrlf inputConverts CRLF to LF on commit without rewriting normal LF files

Safe rule: prefer a repository-level .gitattributes policy first. Local core.autocrlf settings are just a fallback, not a team standard.

Hooks let you run scripts before commit, push, or other Git events. Aliases compress frequent commands into fast shortcuts.

1Useful Aliases
git config --global alias.st status

Advanced Git: Worktree, Submodule, Bisect, and LFS

These commands show up less often, but when you need them, they solve very specific high-value problems.

1Worktree
git worktree add ../project-hotfix hotfix/payment-timeout
git worktree list

Use worktrees when you need two branches checked out at the same time in separate folders.

2Submodule
git submodule add https://github.com/org/shared-ui.git vendor/shared-ui
git submodule update --init --recursive

Submodules pin another repository at a specific commit inside your repository.

3Bisect
git bisect start
git bisect bad
git bisect good v1.4.0

Bisect binary-searches history to find the commit that introduced a bug.

4Git LFS
git lfs install
git lfs track "*.psd"
git add .gitattributes assets/hero.psd
git commit -m "chore: track design binaries with LFS"

LFS keeps large binary files out of normal Git object storage.

Sparse Checkout, Shallow Clone, and Partial Clone

Large repositories do not need to be all-or-nothing. Git has a few high-value techniques for reducing clone time, disk usage, and local noise when you only need part of a codebase.

1Sparse Checkout
git sparse-checkout init --cone
git sparse-checkout set src docs
git sparse-checkout list

Best for monorepos when you need only a few directories but still want normal branch and history behavior.

2Shallow Clone
git clone --depth 1 https://github.com/org/project.git
git fetch --depth 50
git fetch --unshallow

Useful for CI and quick inspection, but weak when you need deep blame, bisect, or full release history.

3Partial Clone
git clone --filter=blob:none https://github.com/org/project.git
cd project
git checkout main

Great for huge repositories because Git downloads history first and fetches large file contents only when they are needed.

TechniqueBest fitMain tradeoff
Sparse checkoutWork on a few folders in a monorepoStill downloads repository history
Shallow cloneCI jobs, demos, and quick read-only setupIncomplete history can break some workflows
Partial cloneHuge repos with heavy binary or generated contentRequires server support and on-demand object fetching

Good default: use sparse checkout when the repository is logically too big for your task, and use shallow or partial clone when network or disk cost is the real problem.

Team Workflows, Branch Protection, and Production Habits

Good Git habits become team policies: small PRs, protected branches, consistent naming, CI checks, and a merge strategy everyone understands.

WorkflowBest fitKey idea
GitHub FlowMost web apps and SaaS teamsBranch from main, open PR, review, merge, deploy often
Trunk-Based DevelopmentHigh-velocity teams with strong CIVery short-lived branches and constant integration into main
Git FlowRelease-heavy products with distinct release and hotfix branchesMore structure, but more overhead too
  • Protect main with required reviews and required status checks.
  • Prefer squash merge for noisy feature branch history unless preserving exact commits matters.
  • Use branch names like feature/cart-discount, fix/login-timeout, or chore/update-eslint.
  • Do not push secrets, generated archives, or local database files.
  • For hotfixes, branch from the currently deployed tag or release branch, not from stale local code.

Git and GitHub Interview Questions with Short Answers

Git is the version control system; GitHub is a hosted platform built around Git repositories and collaboration features.

Merge combines histories and keeps branch structure; rebase rewrites commits so they appear on top of another base, producing a linear history.

Use revert when the bad commit is already shared, because it safely creates a new commit that undoes the change without rewriting shared history.

HEAD is the reference to your current checked-out commit or branch tip.

It temporarily saves uncommitted changes so you can switch context without making a real commit.

It happens when the target branch has not diverged, so Git only moves the branch pointer forward instead of creating a merge commit.
