What is JavaScript?

JavaScript is the language that makes web pages interactive. It started in the browser, but today it also powers servers with Node.js, build tools, automation scripts, desktop apps, and edge functions.

Think of JavaScript as the behavior layer of the web. HTML gives structure, CSS gives style, and JavaScript gives logic, interaction, validation, state updates, and async communication.

Dynamic Language

Flexible syntax, rapid iteration, and easy experimentation make it ideal for both prototypes and production apps.

Runs Everywhere

Browsers, Node.js, serverless functions, desktop apps, and mobile frameworks all depend on modern JavaScript.

Multi-Paradigm

You can write imperative, functional, object-oriented, and event-driven code in the same language.

Huge Ecosystem

React, Vue, Next.js, Express, Vite, Playwright, and thousands of packages sit on top of the language.

Execution Model and Event Loop

JavaScript is single-threaded at the language level. It processes one stack frame at a time, while the browser or runtime helps with timers, network calls, DOM events, and queues.

The important mental model is: call stack for current work, Web APIs / runtime APIs for background work, microtask queue for promise jobs, and task queue for timers and events.

Call Stack

Functions run here. If the stack is busy, nothing else can execute.

Runtime APIs

Timers, fetch, and DOM events wait outside the stack until they finish.

Microtasks

Promise callbacks run before normal queued tasks after the stack becomes empty.

Tasks

setTimeout, UI events, and message callbacks wait in the task queue.

setTimeout(() => {
  console.log('timeout');
}, 0);

Promise.resolve().then(() => {
  console.log('promise');
});

console.log('end');

// end

this, call, apply, and bind

The value of this depends on how a function is called. Arrow functions do not create their own this; they capture it from the surrounding scope.

const team = {
  name: 'Platform Team',
  printWithFunction() {
    setTimeout(function () {
      console.log(this.name); // undefined in strict mode / window in browser globals
    }, 100);
  },
  printWithArrow() {
    setTimeout(() => {
      console.log(this.name); // Platform Team
    }, 100);
  },
};

function greet(city, country) {
  return `${this.name} from ${city}, ${country}`;
}

const person = { name: 'Asha' };

console.log(greet.call(person, 'Surat', 'India'));
console.log(greet.apply(person, ['Surat', 'India']));

const greetFromSurat = greet.bind(person, 'Surat');
console.log(greetFromSurat('India'));

Rule of thumb: use normal functions for object methods, use arrow functions mostly for callbacks and inline transformations.

Prototypes, Classes, and Inheritance

JavaScript is prototype-based. Classes are a cleaner syntax over that model, but under the hood prototypes still power inheritance and shared behavior.

class Account {
  #balance = 0;

  constructor(owner) {
    this.owner = owner;
  }

  deposit(amount) {
    this.#balance += amount;
    return this;
  }

  get balance() {
    return this.#balance;
  }
}

class SavingsAccount extends Account {
  constructor(owner, interestRate) {
    super(owner);
    this.interestRate = interestRate;
  }

  addInterest() {
    const interest = this.balance * this.interestRate;
    this.deposit(interest);
  }
}

const account = new SavingsAccount('Riya', 0.05);
account.deposit(1000);
account.addInterest();

console.log(account.owner);
console.log(account.balance);

Key point: methods defined on a class live on the prototype, so all instances share the same function definitions.

Promises

A promise represents a value that may arrive later. It can be pending, fulfilled, or rejected. Promises are the foundation for async/await.

const wait = (ms) => new Promise((resolve) => {
  setTimeout(resolve, ms);
});

wait(500)
  .then(() => console.log('Half a second passed'))
  .catch((error) => console.error('Unexpected error', error))
  .finally(() => console.log('Done'));

Promise.allSettled([
  Promise.resolve('first'),
  Promise.reject(new Error('Boom')),
]).then((results) => {
  console.log(results);
});
all

Fail fast when any promise rejects.

allSettled

Wait for every result, success or failure.

race

Use the first settled promise.

any

Resolve when the first promise succeeds.

Async/Await and Fetch API

async/await is syntax built on top of promises. It makes async code read like synchronous code while still being non-blocking.

async function getUsers() {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), 5000);

  try {
    const response = await fetch('https://jsonplaceholder.typicode.com/users', {
      signal: controller.signal,
    });

    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }

    return await response.json();
  } catch (error) {
    console.error('Request failed:', error.message);
    return [];
  } finally {
    clearTimeout(timeoutId);
  }
}

async function init() {
  const users = await getUsers();
  console.log(users.slice(0, 2));
}

init();

Best practice: always check response.ok, wrap awaits that can fail in try/catch, and consider cancellation for slow requests.

2Debounce Utility
function debounce(callback, delay = 300) {
  let timerId;

  return (...args) => {
    clearTimeout(timerId);
    timerId = setTimeout(() => {
      callback(...args);
    }, delay);
  };
}

const search = debounce((value) => {
  console.log('Search API for:', value);
}, 400);

Iterators and Generators

Iterables power for...of, spread syntax, and many built-in APIs. Generators make it easy to create lazy sequences.

const range = {
  start: 1,
  end: 3,
  [Symbol.iterator]() {
    let current = this.start;
    const end = this.end;

    return {
      next() {
        if (current <= end) {
          return { value: current++, done: false };
        }

        return { value: undefined, done: true };
      },
    };
  },
};

function* orderIds() {
  let id = 1;
  while (id <= 3) {
    yield `ORD-${id++}`;
  }
}

Prototype Deep Dive

Every object in JavaScript links to another object through its internal prototype chain. Property lookup walks that chain until it finds a matching key or reaches null.

Own Properties

Values stored directly on the current object, such as instance fields created inside a constructor.

Prototype Lookup

If a property is missing locally, JavaScript checks the prototype, then that prototype's prototype, and so on.

Shared Methods

Methods on a prototype are shared by all instances, which saves memory compared with recreating functions per object.

Object.create

Lets you build an object with an explicit prototype without using class syntax.

1Property Lookup Example
const userMethods = {
  greet() {
    return `Hello ${this.name}`;
  },
  isAdmin() {
    return this.role === 'admin';
  },
};

const user = Object.create(userMethods);
user.name = 'Neel';
user.role = 'editor';

console.log(user.greet());
console.log(user.isAdmin());
console.log(user.hasOwnProperty('name'));   // true
console.log(user.hasOwnProperty('greet'));  // false
console.log(Object.getPrototypeOf(user) === userMethods); // true
2Interview-Style Example: Constructor vs Prototype
function BadUser(name) {
  this.name = name;
  this.sayHi = function () {
    return `Hi ${this.name}`;
  };
}

function BetterUser(name) {
  this.name = name;
}

BetterUser.prototype.sayHi = function () {
  return `Hi ${this.name}`;
};

const a = new BadUser('Asha');
const b = new BadUser('Riya');
const c = new BetterUser('Neel');
const d = new BetterUser('Tina');

console.log(a.sayHi === b.sayHi); // false: new function for each instance
console.log(c.sayHi === d.sayHi); // true: shared prototype method

Interview takeaway: classes are mostly syntax sugar. If asked what JavaScript inheritance really uses under the hood, the answer is the prototype chain.

Currying and Partial Application

Currying transforms a function that takes multiple arguments into a sequence of functions that each take one argument. It is useful when you want reusable, preconfigured functions.

Currying

Turn fn(a, b, c) into fn(a)(b)(c) so arguments can be supplied in stages.

Partial Application

Pre-fill some arguments now and supply the rest later, such as creating a tax calculator or formatter.

Functional Composition

Curried functions are easier to reuse in pipelines because each step accepts fewer, simpler inputs.

1Simple Currying Example
const multiply = (a) => (b) => a * b;

const double = multiply(2);
const triple = multiply(3);

console.log(double(8));
console.log(triple(8));

const formatCurrency = (locale) => (currency) => (value) =>
  new Intl.NumberFormat(locale, { style: 'currency', currency }).format(value);

const formatINR = formatCurrency('en-IN')('INR');
console.log(formatINR(2599));
2Interview-Style Example: Generic curry Helper
function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn(...args);
    }

    return (...nextArgs) => curried(...args, ...nextArgs);
  };
}

function total(a, b, c) {
  return a + b + c;
}

const curriedTotal = curry(total);

console.log(curriedTotal(1)(2)(3));
console.log(curriedTotal(1, 2)(3));
console.log(curriedTotal(1)(2, 3));

Interview takeaway: if asked for the difference, currying changes the function shape, while partial application usually means fixing a few arguments without necessarily turning every call into one-argument functions.

Memory Leaks and Cleanup Patterns

A memory leak happens when code keeps references to values that are no longer useful, which prevents garbage collection from reclaiming them. In browser apps, leaks often come from event listeners, timers, detached DOM nodes, or ever-growing caches.

Unremoved Listeners

Attaching the same handler repeatedly without removing it can keep components and closures alive.

Forgotten Timers

Intervals keep firing until you clear them, even if the UI that created them is gone.

Detached DOM Nodes

Keeping old elements in arrays or maps can retain far more state than expected.

Unbounded Caches

A cache that never expires or prunes entries will keep growing over time.

1Leak Pattern and Fix
function mountWidget() {
  const button = document.querySelector('#saveBtn');
  const clickHistory = [];

  function handleClick() {
    clickHistory.push(Date.now());
    console.log(`Clicked ${clickHistory.length} times`);
  }

  button.addEventListener('click', handleClick);

  const intervalId = setInterval(() => {
    console.log('autosave check');
  }, 5000);

  return function cleanup() {
    button.removeEventListener('click', handleClick);
    clearInterval(intervalId);
  };
}

const cleanupWidget = mountWidget();
// Later, when the widget is removed:
cleanupWidget();
2Interview-Style Example: Why This Leaks
const cachedNodes = [];

function rememberNode(node) {
  cachedNodes.push(node);
}

function renderMessage(text) {
  const item = document.createElement('li');
  item.textContent = text;
  rememberNode(item);
  return item;
}

// If nodes are removed from the DOM but still stored in cachedNodes,
// the array keeps references alive and memory usage can grow.

Interview takeaway: garbage collection only frees memory for unreachable values. If your code keeps a live reference, the runtime cannot reclaim it.

Error Handling and Regex

Good JavaScript code handles invalid input, network failures, and user mistakes gracefully. Regex is also useful for validation and text cleanup when used carefully.

1Custom Errors
class ValidationError extends Error {
  constructor(message) {
    super(message);
    this.name = 'ValidationError';
  }
}

function parseAge(value) {
  const age = Number(value);

  if (Number.isNaN(age) || age < 0) {
    throw new ValidationError('Age must be a positive number');
  }

  return age;
}

try {
  console.log(parseAge('24'));
  console.log(parseAge('-4'));
} catch (error) {
  if (error instanceof ValidationError) {
    console.warn(error.message);
  } else {
    console.error('Unexpected error:', error);
  }
} finally {
  console.log('Validation attempt finished');
}
2Regex Basics

Best Practices and Quick Revision

Once you understand the language features, the next step is writing code that stays readable, testable, and maintainable as a project grows.

Best Practices
  • Prefer const, then use let only when reassignment is necessary.
  • Use strict equality unless you intentionally want coercion.
  • Keep functions small and focused on one responsibility.
  • Avoid mutating arguments when a copied value is clearer.
  • Handle async failures explicitly with try/catch.
  • Use modules to keep files cohesive and dependencies visible.
Interview Quick Notes
  • Closure: function remembers outer variables.
  • Hoisting: declarations are processed before execution.
  • Prototype: shared behavior lookup chain for objects.
  • Promise: future result of async work.
  • Event loop: decides when queued work returns to the stack.
  • this: determined by call-site, not where defined, except arrows.

function createTodoStore() {
  let items = [];

  return {
    add(title) {
      items = [...items, { id: crypto.randomUUID(), title, done: false }];
      return items;
    },
    toggle(id) {
      items = items.map((item) =>
        item.id === id ? { ...item, done: !item.done } : item
      );
      return items;
    },
    all() {
      return items;
    },
  };
}

JavaScript Interview Questions

These are the questions that repeatedly show up in interviews because they reveal whether someone understands the language model, not just the syntax.

var is function-scoped and can be redeclared. let and const are block-scoped. const cannot be reassigned after initialization. In modern code, prefer const by default and use let only when a value must change.

A closure is created when an inner function keeps access to variables from its outer lexical scope even after the outer function has finished. Closures are used in factories, private state, memoization, and event handlers.

JavaScript runs synchronous code on the call stack first. When the stack is empty, the runtime checks the microtask queue first, then the task queue. Promise callbacks run as microtasks, while timers and UI events are generally queued as tasks.

== allows type coercion before comparison, while === compares both type and value directly. Prefer === because it is more predictable and avoids implicit conversion bugs.

A shallow copy duplicates only the first level, so nested objects still share references. A deep copy duplicates nested values too. Spread syntax and Object.assign() create shallow copies. structuredClone() can deep copy many built-in data structures.

Arrow functions have shorter syntax and lexical this. They do not have their own arguments, cannot be used as constructors with new, and are usually best for callbacks. Regular functions are better for object methods and constructor-like patterns.
