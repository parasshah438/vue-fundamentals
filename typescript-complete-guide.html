What is TypeScript?

TypeScript is a superset of JavaScript. You write normal JavaScript syntax plus type annotations, interfaces, generics, and compiler rules. TypeScript then checks your code and compiles it to plain JavaScript that browsers and Node.js can run.

Static Analysis

Catch type mismatches before code reaches the browser or server.

Better Autocomplete

Editors understand object shapes, function signatures, and available APIs.

Safer Refactors

Rename fields and move code with compiler feedback instead of guessing.

Runs as JavaScript

Your deployment output is still JavaScript. Types disappear after compilation.

JavaScript vs TypeScript in one example
JavaScript
function totalPrice(price, taxRate) {
  return price + price * taxRate;
}

totalPrice(100, 0.18);   // 118
totalPrice('100', 0.18); // '10018' -> bug
TypeScript
function totalPrice(price: number, taxRate: number): number {
  return price + price * taxRate;
}

totalPrice(100, 0.18);
// totalPrice('100', 0.18);
// Error: Argument of type 'string' is not assignable to type 'number'.

Why TypeScript?

The short answer: JavaScript gives you flexibility, but TypeScript adds constraints that help teams move faster with fewer runtime surprises.

1
You see bugs during development

Undefined values, wrong function arguments, missing object fields, and impossible code paths become compile-time errors.

2
Large refactors become realistic

If you rename a field or change a return type, the compiler tells you exactly where the contract broke.

3
Tooling gets much smarter

Autocomplete, go-to-definition, rename symbol, and inline docs all become more accurate because the editor knows your shapes.

4
Public APIs become self-documenting

Well-typed functions explain what they expect and what they return without a second README file.

JavaScript TypeScript
Runtime-focused; errors often appear after executionCompiler catches many mistakes before execution
Flexible object shapesExplicit contracts via interfaces and aliases
Refactors rely more on tests and manual searchRefactors are safer with compiler guidance
Good for quick scriptsGreat for quick scripts and large applications
Documentation is often separateTypes themselves become documentation

Learning Roadmap

If you want a practical path from beginner to advanced, follow this order. Each step builds on the previous one.

1. Setup

Install TypeScript, create tsconfig.json, compile a file, and run output with Node.

2. Fundamentals

Primitive types, arrays, objects, functions, unions, interfaces, and narrowing.

3. Advanced Types

Generics, utility types, mapped types, classes, module boundaries, and declaration files.

4. Production Usage

Node APIs, React components, fetch typing, testing, linting, and strict compiler rules.

Focus order
  • Start with strict: true so you learn the correct patterns early.
  • Prefer unknown to any when data comes from outside your app.
  • Learn union types and narrowing before heavy generics.
  • Types disappear at runtime, so still validate API payloads and user input.
Mindset shift

The best TypeScript code usually is not the most complex type gymnastics. It is the code with the clearest contracts. Use types to make business rules obvious, not to impress yourself.

Rule of thumb: if a type takes longer to understand than the runtime code it protects, simplify it.

tsconfig.json Explained

This file controls how TypeScript reads your project, which JavaScript version it targets, which files it includes, and how strict the compiler should be.

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "rootDir": "src",
    "outDir": "dist",
    "sourceMap": true
  },
  "include": ["src/**/*.ts", "src/**/*.tsx"]
}
OptionWhy it matters
strictTurns on the most useful type-safety checks. Leave this on for serious projects.
targetControls which JavaScript syntax TypeScript emits.
moduleSets the module system, such as CommonJS or modern ESM.
moduleResolutionControls how imports are resolved on disk.
rootDir / outDirKeep source files and compiled files separate.
sourceMapMaps compiled output back to original source for debugging.
noUncheckedIndexedAccessMakes array and dictionary access safer by including undefined.
exactOptionalPropertyTypesMakes optional properties behave more precisely.

Good default: start strict, then relax rules only when you know exactly why a rule hurts your project.

Type Basics

These are the building blocks of TypeScript: primitive types, arrays, tuples, literal types, and type inference.

1Primitives and inference
const title = 'TS Basics';   // inferred as string
let score: number = 95;
let published: boolean = true;

// Prefer inference when it is obvious.
const lessonCount = 18;

// Add explicit types when the contract matters.
let price: number;
price = 499;
2Arrays, tuples, and readonly
const tags: string[] = ['types', 'generics'];
const rgb: [number, number, number] = [255, 140, 0];

const points: readonly number[] = [10, 20, 30];
// points.push(40); // Error: readonly array

type Size = 'sm' | 'md' | 'lg';
const buttonSize: Size = 'md';
unknown vs any

any disables type safety. unknown forces you to check before using the value.

let rawValue: unknown = JSON.parse('{"name":"Ada"}');

if (typeof rawValue === 'object' && rawValue !== null) {
  console.log(rawValue);
}
Type assertions

Assertions tell the compiler what you believe to be true. Use them carefully.

const input = document.getElementById('email') as HTMLInputElement | null;

if (input) {
  console.log(input.value);
}

Functions

TypeScript functions can describe parameters, return values, optional arguments, rest arguments, overloads, and callbacks.

1Parameters and return types
function greet(name: string, prefix = 'Hello'): string {
  return `${prefix}, ${name}`;
}

function sum(...values: number[]): number {
  return values.reduce((total, value) => total + value, 0);
}
2Function types and callbacks
type Formatter = (value: string) => string;

function runFormatter(value: string, format: Formatter): string {
  return format(value);
}

const upper = runFormatter('typescript', text => text.toUpperCase());
3Function overloads
function format(input: string): string;
function format(input: number): string;
function format(input: string | number): string {
  if (typeof input === 'number') {
    return `$${input.toFixed(2)}`;
  }

  return input.trim();
}

format(' hello ');
format(49.99);

Objects, Interfaces, and Type Aliases

Objects are where TypeScript becomes most valuable. Describe shapes clearly, especially for API payloads, props, and service contracts.

1Interfaces
interface User {
  id: number;
  name: string;
  email?: string;
  readonly role: 'admin' | 'editor' | 'viewer';
}

const user: User = {
  id: 1,
  name: 'Ada Lovelace',
  role: 'admin'
};
2Type aliases
type ApiStatus = 'idle' | 'loading' | 'success' | 'error';

type Product = {
  sku: string;
  price: number;
  tags: string[];
};

type ProductMap = Record<string, Product>;
3When to use which?
Use casePreferReason
Public object contractinterfaceReadable and extendable.
Union or intersectiontypeAliases handle composition very well.
Function signatureEitherChoose the clearer style for the team.
Mapped or conditional typetypeAdvanced type expressions require aliases.

Narrowing and Type Guards

TypeScript often starts with a broad type like string | number or unknown. Narrowing is how you prove which branch is active.

1Built-in guards
function printId(id: string | number) {
  if (typeof id === 'string') {
    console.log(id.toUpperCase());
    return;
  }

  console.log(id.toFixed(0));
}
2Custom predicate
type User = { id: number; name: string };

function isUser(value: unknown): value is User {
  return typeof value === 'object'
    && value !== null
    && 'id' in value
    && 'name' in value;
}
3Discriminated unions
type LoadState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: string[] }
  | { status: 'error'; message: string };

function render(state: LoadState) {
  switch (state.status) {
    case 'idle':
      return 'Waiting';
    case 'loading':
      return 'Loading...';
    case 'success':
      return state.data.join(', ');
    case 'error':
      return state.message;
  }
}

Generics

Generics let you write reusable logic while preserving the specific type flowing through it.

1Generic functions
function firstItem<T>(items: T[]): T | undefined {
  return items[0];
}

const firstName = firstItem(['Ada', 'Grace']);
const firstNumber = firstItem([10, 20, 30]);
2Generic interfaces
interface ApiResponse<T> {
  success: boolean;
  data: T;
  message?: string;
}

type User = { id: number; name: string };

const response: ApiResponse<User[]> = {
  success: true,
  data: [{ id: 1, name: 'Ada' }]
};
3Constraints
function getId<T extends { id: number }>(value: T): number {
  return value.id;
}

getId({ id: 42, name: 'Compiler' });
// getId({ name: 'Missing id' }); // Error

Utility and Meta Types

Utility types save you from rewriting the same object transformations over and over.

TypeWhat it doesShort example
Partial<T>Makes every property optional.Partial<User>
Required<T>Makes every property required.Required<Settings>
Pick<T, K>Selects a subset of keys.Pick<User, 'id' | 'name'>
Omit<T, K>Removes selected keys.Omit<User, 'password'>
Record<K, V>Creates a key-value object type.Record<string, number>

Patch objects with Partial
interface User {
  id: number;
  name: string;
  email: string;
}

function updateUser(id: number, patch: Partial<User>) {
  return { id, ...patch };
}
Mapped types
type Flags<T> = {
  [K in keyof T]: boolean;
};

type FeaturePermissions = Flags<{
  darkMode: string;
  exportCsv: string;
}>;

Classes and OOP

Classes are useful when you need instances, state, methods, inheritance, or framework-level class APIs. TypeScript adds strong typing to all of it.

1Class basics
class Wallet {
  constructor(private balance = 0) {}

  deposit(amount: number) {
    this.balance += amount;
  }

  get currentBalance(): number {
    return this.balance;
  }
}
2Interfaces + abstract classes
interface Logger {
  log(message: string): void;
}

abstract class BaseService implements Logger {
  abstract log(message: string): void;
}

class ConsoleService extends BaseService {
  log(message: string) {
    console.log(message);
  }
}

Pragmatic advice: use classes when they model stateful objects clearly. For small reusable logic, plain functions plus typed objects are often simpler.

Modules, Imports, and Declaration Files

Modern TypeScript projects use ES modules. Declaration files are used to describe types for libraries or globals when TypeScript cannot infer them automatically.

1Exports and imports
// math.ts
export function add(a: number, b: number): number {
  return a + b;
}

export type Currency = 'USD' | 'EUR';

// app.ts
import { add, type Currency } from './math';

const total = add(10, 5);
const currency: Currency = 'USD';
2Declaration files
// env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    DATABASE_URL: string;
    APP_ENV: 'development' | 'production';
  }
}

// global.d.ts
declare const APP_VERSION: string;
Namespaces?

Namespaces still exist in TypeScript, but for modern apps you almost always want modules with import and export instead.

Async Code, APIs, and JSON

Asynchronous code is where runtime uncertainty meets compile-time types. The key is to type promises clearly and validate unknown external data.

1Typed async functions
type User = { id: number; name: string };

async function fetchUser(id: number): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  const data = (await response.json()) as User;
  return data;
}
2Promise.all
const [user, posts] = await Promise.all([
  fetchUser(1),
  fetch('/api/posts').then(res => res.json() as Promise<string[]>)
]);

console.log(user.name, posts.length);
3Important rule: types disappear at runtime
type User = { id: number; name: string };

function parseUser(value: unknown): User {
  if (
    typeof value === 'object' &&
    value !== null &&
    'id' in value &&
    'name' in value
  ) {
    return value as User;
  }

  throw new Error('Invalid user payload');
}

DOM, Events, and Forms

Browser work benefits a lot from TypeScript because the DOM API is large and easy to misuse.

1Typed form event
const form = document.querySelector<HTMLFormElement>('#signup-form');

form?.addEventListener('submit', event => {
  event.preventDefault();
  const data = new FormData(event.currentTarget as HTMLFormElement);
  console.log(data.get('email'));
});
2Safe element access
const button = document.querySelector<HTMLButtonElement>('#save-btn');

if (button) {
  button.disabled = true;
  button.textContent = 'Saving...';
}
Tip

Prefer selector generics like querySelector<HTMLInputElement>() over a blind assertion when possible. They keep the intent explicit.

Node, React, and Testing

TypeScript is most useful when it describes boundaries between parts of your application: services, components, APIs, and tests.

Node Services

Type environment variables, service inputs, and returned domain objects.

React Components

Props, event handlers, API hooks, and state become easier to reason about.

Tests

Typed tests catch misuse of helpers, mocks, and component contracts.

1React props example
type ButtonProps = {
  label: string;
  disabled?: boolean;
  onClick: () => void;
};

export function Button({ label, disabled = false, onClick }: ButtonProps) {
  return (
    <button disabled={disabled} onClick={onClick}>
      {label}
    </button>
  );
}
2Vitest example
import { describe, expect, it } from 'vitest';

function add(a: number, b: number) {
  return a + b;
}

describe('add', () => {
  it('returns the total', () => {
    expect(add(2, 3)).toBe(5);
  });
});
3Node service pattern
type Order = {
  id: string;
  total: number;
  status: 'pending' | 'paid';
};

class OrderService {
  markPaid(order: Order): Order {
    return { ...order, status: 'paid' };
  }
}

Common TypeScript Errors and How to Think About Them

Most compiler errors are not random. They are TypeScript telling you that your code contract is unclear, incomplete, or impossible.

Error shapeWhat it usually meansTypical fix
Type 'string | undefined' is not assignable...A value may be missing.Check it, provide a default, or change the target type.
Object is possibly 'null'You are using a DOM or API value without checking it exists.Narrow with if (value) first.
Property does not exist on type ...You used a property before proving the object shape.Use narrowing, a better type, or a predicate function.
Argument of type 'X' is not assignable to 'Y'You passed the wrong shape into a function.Align the caller with the function contract.
Type 'never'...Your narrowing reached an impossible branch or a union is fully exhausted.Review the union and switch branches.
 
type User = { email?: string };

function sendEmail(user: User) {
  if (!user.email) {
    throw new Error('Missing email');
  }

  // From here, TypeScript knows email is string.
  console.log(user.email.toLowerCase());
}

Debug flow: read the full compiler message, inspect the source type, inspect the target type, and then decide whether the contract or the data is wrong.

Best Practices

Do this
  • Keep strict: true enabled.
  • Model states with unions instead of booleans and magic strings.
  • Use interfaces or aliases to document domain models clearly.
  • Validate API input and JSON data at runtime.
  • Run tsc --noEmit in CI even when you use a fast bundler.
Avoid this
  • Do not silence errors with any unless you truly have no better boundary.
  • Do not use type assertions as a substitute for real checks.
  • Do not write clever generic types when a simpler contract is enough.
  • Do not ignore compiler warnings and assume tests will catch everything.
  • Do not forget that browser and Node environments expose different globals and libs.
Practical project checklist
  • Use src/ and dist/ separation.
  • Add typecheck, lint, and test scripts.
  • Keep environment variable types in env.d.ts.
  • Prefer precise unions over generic string when values are finite.
  • Use runtime validation at API boundaries.
  • Document tricky types with examples, not only comments.
