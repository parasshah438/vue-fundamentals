Yii2 Complete Guide for Real Projects

This guide follows the same tutorial style as the Next.js page, but the topic is Yii. It covers Yii2 from installation to deployment, with practical examples for MVC, ActiveRecord, forms, REST, queue jobs, testing, and production work. It also includes a dedicated Yii3 section so you can understand what changed in the newer generation.

A clear map of how Yii apps are structured, extended, secured, and deployed.

  • Understand the Yii2 basic and advanced templates.
  • Learn routing, controllers, models, validation, and views.

What is Yii?

Yii is a high-performance PHP framework for building web applications, admin panels, APIs, and full business systems. Yii2 is the mature, production-proven version used in many long-running projects. Its strengths are MVC structure, strong code generation tooling, solid caching support, flexible configuration, and tight integration with database-centric apps.

Fast start

Composer templates, Gii, migrations, and conventions let you move from zero to CRUD quickly.

Strong MVC

Controllers, form models, AR models, views, layouts, and modules are clearly separated.

DB friendly

ActiveRecord, Query Builder, migration tooling, and relational helpers are first-class.

Enterprise ready

RBAC, caching, queue integration, console commands, and testing support fit larger systems.

Why Yii2?

Choose Yii2 when you want a pragmatic PHP framework with batteries included, especially for back-office systems, dashboards, CMS-style tools, internal enterprise apps, and APIs backed by relational databases.

NeedHow Yii2 helpsExample
Rapid admin workGii generates models, CRUD, and forms fast.Product, order, customer, and invoice management.
Business rulesForm models, validators, behaviors, and services keep rules organized.Loan approval, HR approvals, ecommerce back-office logic.
PerformanceFragment caching, data caching, eager loading, and asset optimization are built in.Large catalog pages or analytics dashboards.
Long-lived codebaseModules, components, console commands, and RBAC scale better than ad hoc PHP code.ERP, CRM, booking, and logistics systems.

Yii2 vs Yii3 at a Glance

Yii2 is the stable and widely deployed generation. Yii3 is the newer direction with a more package-based architecture and stronger PSR alignment. Yii3 is not a drop-in replacement for Yii2, so treat it as a separate architecture choice rather than a simple version bump.

AreaYii2Yii3
ArchitectureClassic integrated framework experienceMore package-based and modular
StyleMVC with strong conventionsPSR-friendly, middleware-oriented, package-first
AdoptionProduction-proven and common in legacy plus active appsNewer ecosystem; evaluate package maturity per use case
MigrationExisting apps stay on Yii2 unless there is a real business reason to moveBest for new builds or selective package adoption

Structure reading tip:

`controllers/` handles incoming requests, `models/` holds data and validation logic, `views/` renders HTML, `config/` wires the app, and `commands/` is for console automation.

Request Lifecycle

Understanding the request flow makes debugging much easier than blindly changing controllers and hoping for the best.

1. Entry script

The request enters through `web/index.php` and boots the application.

2. Router

URL rules resolve the route to a controller action.

3. Action logic

Models validate input, query data, and prepare a response.

4. Response

A view, JSON payload, redirect, or error response is returned.

Routing and URLs

Yii routing is configured through URL rules. Pretty URLs should usually be enabled early so route names stay clean and stable.

// config/web.php
'components' => [
    'urlManager' => [
        'enablePrettyUrl' => true,
        'showScriptName' => false,
        'rules' => [
            'posts' => 'post/index',
            'post/<id:\d+>' => 'post/view',
        ],
    ],
],

That rule maps `/post/5` to `PostController::actionView($id)`.

Controllers and Actions

Controllers receive requests and coordinate work. Keep them thin by moving validation, business rules, and persistence into models or services.

namespace app\controllers;

use Yii;
use app\models\ContactForm;
use yii\web\Controller;

class SiteController extends Controller
{
    public function actionContact()
    {
        $model = new ContactForm();

        if ($model->load(Yii::$app->request->post()) && $model->validate()) {
            // send mail or save lead
            Yii::$app->session->setFlash('success', 'Message sent');
            return $this->refresh();
        }

        return $this->render('contact', ['model' => $model]);
    }
}

Views and Layouts

Views are PHP templates. Layouts wrap repeated page structure around the per-page content. Yii helpers keep HTML generation cleaner and safer.

<h1><?= Html::encode($model->title) ?></h1>

<!doctype html>

Models and Forms

In Yii, not every model is a database table. Form models are useful for contact forms, search forms, login forms, and workflows that should validate input without writing directly to the database.

namespace app\models;

use yii\base\Model;

class ContactForm extends Model
{
    public $name;
    public $email;
    public $message;

    public function rules(): array
    {
        return [
            [['name', 'email', 'message'], 'required'],
            ['email', 'email'],
            ['message', 'string', 'min' => 10],
        ];
    }
}

Validation Rules

Validation in Yii2 is declarative. Rules live in one place, can be reused in forms and AR models, and support custom methods when built-in validators are not enough.

public function rules(): array
{
    return [
        ['price', 'number', 'min' => 0],
        ['slug', 'match', 'pattern' => '/^[a-z0-9-]+$/'],
        ['start_date', 'validateStartDate'],
    ];
}

public function validateStartDate($attribute): void
{
    if (strtotime($this->$attribute) < time()) {
        $this->addError($attribute, 'Start date must be in the future.');
    }
}

Yii keeps DB configuration in a component, usually `config/db.php`. Once configured, the app can use ActiveRecord, Query Builder, migrations, and transactions against the same connection.

ActiveRecord

ActiveRecord maps rows to PHP objects. It is fast for normal CRUD work and easy for teams to understand, especially when models stay focused and relations are explicit.

namespace app\models;

use yii\db\ActiveRecord;

class Post extends ActiveRecord
{
    public static function tableName(): string
    {
        return '{{%post}}';
    }
}

$post = new Post();
$post->title = 'First Post';
$post->body = 'Yii2 makes CRUD straightforward.';
$post->save();

Relations

Relations express the data graph and make eager loading possible. Use them carefully to prevent N+1 query problems.

class Post extends ActiveRecord
{
    public function getAuthor()
    {
        return $this->hasOne(User::class, ['id' => 'user_id']);
    }

    public function getComments()
    {
        return $this->hasMany(Comment::class, ['post_id' => 'id']);
    }
}

$posts = Post::find()->with(['author', 'comments'])->all();

Migrations

Treat the schema as code. Migrations keep database structure reviewable and reproducible across environments.

php yii migrate/create create_post_table
php yii migrate
use yii\db\Migration;

class m260427_100000_create_post_table extends Migration
{
    public function safeUp()
    {
        $this->createTable('{{%post}}', [
            'id' => $this->primaryKey(),
            'title' => $this->string()->notNull(),
            'body' => $this->text()->notNull(),
            'created_at' => $this->integer()->notNull(),
        ]);
    }

    public function safeDown()
    {
        $this->dropTable('{{%post}}');
    }
}

Gii CRUD Generator

Gii is one of Yii2's most productive features. It can generate models, forms, CRUD pages, and more. Use it to accelerate scaffolding, then review and refactor the generated code instead of shipping it unchanged.

composer require --prefer-dist yiisoft/yii2-gii --dev
php yii gii

Best use: let Gii generate the first 60 percent of admin CRUD, then move domain logic out of generated controllers when the project becomes serious.

Components and Helpers

Custom components keep cross-cutting logic out of controllers. Yii helpers such as `Html`, `ArrayHelper`, and `Url` reduce repetition and improve safety.

namespace app\components;

use yii\base\Component;

class InvoiceFormatter extends Component
{
    public function formatCurrency(float $amount): string
    {
        return number_format($amount, 2) . ' USD';
    }
}

// config/web.php
'components' => [
    'invoiceFormatter' => app\components\InvoiceFormatter::class,
],

Dependency Injection Container

Yii2 includes a DI container that can instantiate classes, resolve dependencies, and bind interfaces. This matters when controllers should depend on services rather than raw implementation details.

Yii::$container->set(app\services\MailerInterface::class, app\services\SmtpMailer::class);

$service = Yii::$container->get(app\services\ReportService::class);

Events and Behaviors

Events let you react to lifecycle hooks, while behaviors attach reusable logic to models or components. They are powerful, but overuse can hide business logic if events are scattered without discipline.

$model->on(Post::EVENT_AFTER_INSERT, function ($event) {
    Yii::info('New post created: ' . $event->sender->id);
});
use yii\behaviors\TimestampBehavior;

public function behaviors(): array
{
    return [TimestampBehavior::class];
}

Modules

Modules help split larger apps into domain-focused sections such as admin, billing, support, or API. They reduce controller sprawl and make large projects easier to navigate.

        'class' => app\modules\admin\Module::class,

Authentication and RBAC

Yii2 supports identity-based login flows and role-based permissions. For business apps, put actual authorization rules behind RBAC or service-level permission checks, not only in buttons or menus.

Authentication

Use `IdentityInterface`, login forms, and auth methods like sessions or bearer tokens.

Authorization

Use RBAC roles and permissions so access decisions stay centralized.

public function behaviors(): array
{
    return [
        'access' => [
            'class' => yii\filters\AccessControl::class,
            'only' => ['create', 'update', 'delete'],
            'rules' => [
                [
                    'allow' => true,
                    'roles' => ['@'],
                ],
            ],
        ],
    ];
}

Cache and Session

Caching is not optional when dashboards or catalog pages start doing repeated heavy queries. Yii2 supports file cache, Redis, Memcached, fragment caching, and dependency-based invalidation.

Queue and Cron

Move slow or retryable work out of web requests. Email sending, webhook retries, PDF generation, imports, and inventory sync are typical queue jobs.

composer require yiisoft/yii2-queue
php yii queue/listen
php yii queue/run
# Windows Task Scheduler or Linux cron example
* * * * * php /var/www/app/yii queue/run

Testing

Yii projects often use Codeception for unit, functional, and acceptance tests. The real value is not the tool name. It is that your business rules stay protected when features evolve.

vendor/bin/codecept build
vendor/bin/codecept run
vendor/bin/codecept run unit
vendor/bin/codecept run functional
public function testPriceCannotBeNegative(): void
{
    $model = new Product(['price' => -1]);
    $this->assertFalse($model->validate());
}

Security

Yii helps with CSRF, XSS-safe encoding, authentication, and access rules, but the framework does not remove the need for careful engineering.

  • Always encode untrusted output with `Html::encode()`.
  • Use validation rules and model scenarios for input handling.
  • Prefer parameter binding and AR over raw string SQL concatenation.
  • Restrict upload types and store uploaded files outside predictable public paths when needed.
  • Do authorization checks in code paths, not only in the UI.

Performance

Performance work in Yii2 usually comes down to query quality, cache discipline, asset strategy, and production PHP configuration.

AreaCommon problemFix
DB queriesN+1 relation loadingUse `with()` or `joinWith()` where appropriate.
RenderingRepeated expensive widgetsUse fragment caching or data caching.
AssetsToo many JS/CSS requestsBundle wisely and use production assets.
PHP runtimeSlow cold starts and repeated file parsingEnable OPcache in production.

Deployment

Production deployment is mostly disciplined PHP deployment: correct environment config, optimized Composer install, generated assets, DB migrations, cache flushing, and web-server configuration.

composer install --no-dev --optimize-autoloader
php yii migrate --interactive=0
php yii cache/flush-all

Deploy order: code, environment file, dependencies, migrations, caches, queue workers, smoke test. Reversing this order creates avoidable downtime.

Deployment Expansion: Shared Hosting, cPanel, VPS, Nginx, and Queue Worker Supervision

The general deployment rules above stay the same, but hosting style changes the exact operational steps. Yii2 can run on cheap shared hosting, on cPanel-managed servers, and on full VPS stacks. The main difference is how much control you have over document root, process management, cron, and long-running queue workers.

Shared Hosting

Lowest control. Best for simple Yii2 apps with minimal background processing.

cPanel

Still managed hosting, but usually with cron, document-root controls, and easier PHP version switching.

VPS

Full server control. Best for serious apps, APIs, queues, and custom infrastructure.

Worker Supervision

Required when queue listeners must stay alive after crashes or reboots.

1Shared Hosting Checklist

Shared hosting works best for the Yii2 basic template or for advanced-template deployments where you can clearly isolate the public web root. The main constraint is that you often cannot run long-lived workers or install system packages.

  • Point the public document root to the Yii `web/` directory, not the project root.
  • Keep `runtime/` and `web/assets/` writable by the web-server user.
  • Disable debug and use production config values.
  • If shell access is limited, run Composer locally or in CI and upload the built project carefully.
  • Prefer cron-driven `php yii queue/run` over `queue/listen` because shared hosting usually kills long-running processes.
2cPanel Deployment Pattern

cPanel is still shared-style hosting, but it usually gives you enough control for proper cron jobs, PHP version selection, and a cleaner deployment process.

public_html/
  index.php           # points into your Yii app web root if required by host setup
  assets/

yii-app/
  commands/
  config/
  models/
  runtime/
  vendor/
  views/
  web/
  • Set the domain document root to `web/` when cPanel allows custom docroots.
  • If the host forces `public_html`, copy or adapt the Yii front controller carefully so only public assets live there.
  • Create cron jobs in cPanel for migrations or queue processing only when those commands are safe and idempotent.
  • Store secrets outside public paths and confirm `.env` or config files are not directly downloadable.
*/2 * * * * /usr/local/bin/php /home/account/yii-app/yii queue/run >> /home/account/logs/queue.log 2>&1
3VPS Deployment Flow

A VPS is the right place for production Yii apps that need Redis, workers, full Nginx control, SSL management, or deployment automation. This is where Yii2 feels most comfortable operationally.

sudo apt update
sudo apt install nginx php-fpm php-mysql php-intl php-mbstring php-xml unzip git supervisor
cd /var/www
git clone your-repo yii-app
cd yii-app
composer install --no-dev --optimize-autoloader
php yii migrate --interactive=0
php yii cache/flush-all

Recommended production stack:

Nginx + PHP-FPM + Composer install + DB migrations + Redis when needed + Supervisor for queue listeners. That gives you far more predictable behavior than trying to force worker-style operations into shared hosting.

4Nginx Configuration Example

With Yii2, the important rule is that Nginx should serve only the `web/` directory and pass PHP requests through the front controller correctly.

server {
    listen 80;
    server_name example.com;
    root /var/www/yii-app/web;
    index index.php;

    location / {
        try_files $uri $uri/ /index.php?$args;
    }

    location ~ \.php$ {
        include snippets/fastcgi-php.conf;
        fastcgi_pass unix:/run/php/php8.2-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
    }

    location ~* /(?:runtime|tests|vendor)/ {
        deny all;
    }
}

For the advanced template, the `root` changes depending on whether you are exposing `frontend/web` or `backend/web`.

5Queue Worker Supervision

If your app sends emails, processes imports, retries webhooks, or generates reports, the queue worker must stay alive. That means you should supervise `queue/listen` with a process manager instead of manually starting it in SSH and hoping it survives.

Use cron when
  • The host does not allow persistent workers.
  • Jobs are light and can tolerate minute-level delay.
  • You only need `queue/run` periodically.
Use supervision when
  • The app needs near-real-time background work.
  • The worker must restart automatically on crash.
  • You control the VPS or dedicated server.
[program:yii-queue]
command=/usr/bin/php /var/www/yii-app/yii queue/listen --verbose=1 --color=0
directory=/var/www/yii-app
autostart=true
autorestart=true
stopasgroup=true
killasgroup=true
user=www-data
numprocs=1
redirect_stderr=true
stdout_logfile=/var/log/supervisor/yii-queue.log
sudo supervisorctl reread
sudo supervisorctl update
sudo supervisorctl status yii-queue
sudo supervisorctl restart yii-queue

Operational rule: shared hosting and most cPanel plans should stick to cron-based `queue/run`. VPS and dedicated servers should prefer supervised `queue/listen` for reliable background processing.

Yii3 New Version Details

Yii3 represents a structural shift rather than just a routine framework update. It is more package-first, more PSR-friendly, and more explicit about middleware, dependency injection, and decoupled components. For teams starting a new system, Yii3 is the part to study. For existing Yii2 systems, the right move is usually to understand Yii3 concepts first, not to rush a rewrite.

What changed
  • More package-based architecture
  • Stronger PSR alignment
  • Middleware-oriented HTTP handling
  • Cleaner separation between packages
Why it matters
  • Better interoperability with the wider PHP ecosystem
  • More selective adoption of only the pieces you need
  • Less coupling to a single monolithic runtime style
Practical caution
  • Yii3 is not a drop-in `composer update` for Yii2
  • Existing Yii2 apps should migrate deliberately, not emotionally
  • Package maturity should be checked per feature area
1Typical Yii3 Skeleton Flow
composer create-project yiisoft/app app
cd app
composer require yiisoft/router yiisoft/validator yiisoft/db-mysql
2Conceptual Differences from Yii2
TopicYii2 mindsetYii3 mindset
HTTP flowController-oriented web applicationMiddleware and handler pipeline
Framework shapeIntegrated framework distributionComposable packages
Dependency injectionAvailable and usefulMore central to the architecture
Migration strategyStable base for existing projectsBetter considered for new builds or selective package adoption

Recommendation: if your team already runs a healthy Yii2 app, improve architecture inside Yii2 first. If you are planning a fresh greenfield system and want a more package-based modern PHP style, evaluate Yii3 early.

Yii2 CRUD Mini Project: Product Manager

This mini project ties the Yii2 basics together into one realistic slice: a Product Manager for an admin panel. It includes a database table, migration, ActiveRecord model, controller actions, views, and a Gii shortcut flow. The goal is not just to see isolated snippets, but to understand how the full CRUD chain works end to end.

Project goal
  • Create products with name, slug, price, status, and description.
  • List products in an admin table.
  • Edit and delete records safely.
  • Use one migration and one ActiveRecord model as the base.
Database shape
  • `id` primary key
  • `name` required
  • `slug` unique
  • `price` decimal
  • `status` active or draft
What you practice
  • Migration authoring
  • Model rules and labels
  • Controller CRUD actions
  • Form and table views
  • Manual code vs Gii-generated scaffolding
1Create the Database Table with a Migration

Start by creating the schema in a migration so the project can be reproduced on every environment.

php yii migrate/create create_product_table
php yii migrate
use yii\db\Migration;

class m260427_120000_create_product_table extends Migration
{
    public function safeUp()
    {
        $this->createTable('{{%product}}', [
            'id' => $this->primaryKey(),
            'name' => $this->string(150)->notNull(),
            'slug' => $this->string(180)->notNull()->unique(),
            'price' => $this->decimal(10, 2)->notNull()->defaultValue(0),
            'status' => $this->smallInteger()->notNull()->defaultValue(1),
            'description' => $this->text(),
            'created_at' => $this->integer()->notNull(),
            'updated_at' => $this->integer()->notNull(),
        ]);
    }

    public function safeDown()
    {
        $this->dropTable('{{%product}}');
    }
}
2Build the Product Model

The model defines the table name, validation rules, labels, and any helper methods your views or controllers need.

namespace app\models;

use yii\behaviors\TimestampBehavior;
use yii\db\ActiveRecord;

class Product extends ActiveRecord
{
    public static function tableName(): string
    {
        return '{{%product}}';
    }

    public function rules(): array
    {
        return [
            [['name', 'slug', 'price'], 'required'],
            [['description'], 'string'],
            [['price'], 'number', 'min' => 0],
            [['status'], 'integer'],
            [['name'], 'string', 'max' => 150],
            [['slug'], 'string', 'max' => 180],
            [['slug'], 'unique'],
            ['slug', 'match', 'pattern' => '/^[a-z0-9-]+$/'],
        ];
    }

    public function attributeLabels(): array
    {
        return [
            'name' => 'Product Name',
            'slug' => 'URL Slug',
            'price' => 'Unit Price',
        ];
    }

    public function behaviors(): array
    {
        return [TimestampBehavior::class];
    }
}
3Create the Controller

The controller is responsible for listing products, showing the form, saving data, and handling updates or deletes. This is the classic CRUD shape every Yii2 team should be comfortable with.

namespace app\controllers;

use app\models\Product;
use yii\data\ActiveDataProvider;
use yii\filters\VerbFilter;
use yii\web\Controller;
use yii\web\NotFoundHttpException;
use Yii;

class ProductController extends Controller
{
    public function behaviors(): array
    {
        return [
            'verbs' => [
                'class' => VerbFilter::class,
                'actions' => [
                    'delete' => ['POST'],
                ],
            ],
        ];
    }

    public function actionIndex()
    {
        $dataProvider = new ActiveDataProvider([
            'query' => Product::find()->orderBy(['id' => SORT_DESC]),
            'pagination' => ['pageSize' => 10],
        ]);

        return $this->render('index', [
            'dataProvider' => $dataProvider,
        ]);
    }

    public function actionCreate()
    {
        $model = new Product();

        if ($model->load(Yii::$app->request->post()) && $model->save()) {
            Yii::$app->session->setFlash('success', 'Product created successfully.');
            return $this->redirect(['index']);
        }

        return $this->render('create', ['model' => $model]);
    }

    public function actionUpdate($id)
    {
        $model = $this->findModel($id);

        if ($model->load(Yii::$app->request->post()) && $model->save()) {
            Yii::$app->session->setFlash('success', 'Product updated successfully.');
            return $this->redirect(['index']);
        }

        return $this->render('update', ['model' => $model]);
    }

    public function actionDelete($id)
    {
        $this->findModel($id)->delete();
        Yii::$app->session->setFlash('success', 'Product deleted successfully.');
        return $this->redirect(['index']);
    }

    protected function findModel($id): Product
    {
        if (($model = Product::findOne($id)) !== null) {
            return $model;
        }

        throw new NotFoundHttpException('The requested product does not exist.');
    }
}

5Manual Flow End to End

How the request moves:

The migration creates the `product` table. The `Product` model maps to that table and validates input. The `ProductController` loads post data and saves the model. The form view renders input fields, and the index view renders a GridView table with edit and delete actions.

Yii Interview Questions with Short Answers

An ActiveRecord model maps to a database table. A form model is for input validation and workflows that may not directly save to a table.

They keep schema changes versioned, reviewable, and reproducible across local, staging, and production environments.

Use Gii for fast scaffolding. Stop relying on generated code once domain rules or architectural boundaries become important and require manual design.

Define clear relations and eager load them using `with()` or `joinWith()` when listing related data.

Authentication proves identity. RBAC decides which actions that authenticated identity is allowed to perform.

Yii3 changes the architecture significantly toward package-based and PSR-friendly design, so migration is more like re-architecture than patching one dependency version.
