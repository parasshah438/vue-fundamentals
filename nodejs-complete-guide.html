What is Node.js?

Node.js is a JavaScript runtime built on the V8 engine. It lets you run JavaScript outside the browser, especially for servers, APIs, command-line tools, background jobs, automation scripts, realtime systems, and build tooling.

V8 Engine

Executes JavaScript fast and keeps the syntax familiar for front-end developers.

Event-Driven I/O

Handles lots of network or file operations efficiently without blocking the whole process.

npm Ecosystem

Ships with the biggest JavaScript package ecosystem for servers, tooling, CLIs, and frameworks.

CLI Friendly

Excellent for scripts, task runners, build tools, and package publishing workflows.

Important: Node.js is not a framework. It is the runtime. Express, Fastify, NestJS, Socket.IO, Prisma, and many other tools run on top of it.

Why Learn Node.js

Node.js is strong when your application spends most of its time waiting on I/O: databases, APIs, message queues, files, sockets, or user requests. It is also a common choice when one team wants to share JavaScript and TypeScript across front end, backend, tooling, and serverless functions.

Best fitWhy it worksWatch out for
REST APIs and web appsFast development loop, huge ecosystem, great JSON handling.Do not block the event loop with CPU-heavy work.
Realtime appsWebSockets, Socket.IO, and event-driven programming feel natural.Connection scaling still needs proper infrastructure.
Background jobs and automationEasy HTTP integrations, queues, and scheduled scripts.Long jobs need retries, logging, and monitoring.
CLI tools and internal toolingNode is already present on many developer machines.Keep package dependencies lean for fast installs.
  • Choose Node.js when your team values fast iteration, APIs, SSR, or shared TypeScript types.
  • Avoid pushing CPU-bound image processing, ML inference, or huge synchronous loops onto the main thread.
  • For CPU-heavy work, use worker threads, queues, native services, or a better-suited runtime.

Node.js Runtime Architecture

A lot of Node confusion disappears when you understand three parts: V8 executes JavaScript, libuv manages asynchronous I/O and the event loop, and the operating system handles actual network and filesystem work.

console.log('start');

setTimeout(() => {
  console.log('timer');
}, 0);

Promise.resolve().then(() => {
  console.log('promise microtask');
});

process.nextTick(() => {
  console.log('nextTick');
});

console.log('end');

Typical output: start, end, nextTick, promise microtask, timer.

Mental model
  • JavaScript runs on the main thread.
  • I/O gets delegated out, then callbacks get queued.
  • process.nextTick() runs before normal microtasks and can starve the loop if abused.
  • CPU-heavy work blocks everything else unless you move it off the main thread.

Cheap performance rule: if one request does heavy synchronous work for 500ms, every other request sharing that process waits too. That is the root reason event-loop awareness matters.

  • Prefer explicit exports over huge utility barrels when the codebase grows.

Async Patterns: Callbacks, Promises, and async or await

Modern Node.js development should default to promises and async or await. You still need to recognize callbacks because core APIs and older packages may expose them.

import fs from 'node:fs/promises';

async function loadConfig() {
  const raw = await fs.readFile('./config.json', 'utf8');
  return JSON.parse(raw);
}

const [config, users] = await Promise.all([
  loadConfig(),
  fetch('https://jsonplaceholder.typicode.com/users').then(r => r.json())
]);

console.log(config.appName, users.length);
Practical async rules
  • Use Promise.all() for independent work that should run in parallel.
  • Use Promise.allSettled() when some failures are acceptable.
  • Always wrap route handlers and job processors with proper error handling.
  • Never forget await on writes that must complete before returning.

Event Loop, Timers, Microtasks, and nextTick

Interviews and production debugging both return to the same question: why did one callback run before another? In Node, that usually comes down to microtasks, timers, process.nextTick(), and libuv phases.

MechanismTypical useRisk
Promise.thenMicrotask after current call stackToo many microtasks can delay timers.
process.nextTickInternal or immediate follow-up workCan starve the event loop if abused in loops.
setTimeoutDelay workDelay is a minimum, not a guarantee.
setImmediateRun after I/O callbacks in many casesOften misunderstood as "faster timeout".

Interview-safe answer: Node is single-threaded for JavaScript execution, not single-threaded for all work. libuv and the OS still handle asynchronous operations underneath.

Build a Raw HTTP Server Before You Reach for a Framework

Express is more productive for most APIs, but a tiny raw HTTP server teaches the request-response model, headers, JSON output, and route branching clearly.

import http from 'node:http';

const server = http.createServer((req, res) => {
  if (req.url === '/health') {
    res.writeHead(200, { 'content-type': 'application/json' });
    res.end(JSON.stringify({ ok: true, uptime: process.uptime() }));
    return;
  }

  res.writeHead(404, { 'content-type': 'application/json' });
  res.end(JSON.stringify({ message: 'Not found' }));
});

server.listen(3000, () => {
  console.log('HTTP server listening on http://localhost:3000');
});

Why this matters: frameworks hide boilerplate, but they do not replace understanding. Knowing raw Node HTTP makes middleware, reverse proxies, and debugging much easier.

Routing, Middleware, Validation, and Error Handling

Most backend bugs are not about the transport layer. They come from missing validation, weak authorization, or inconsistent error handling. Route design should reflect business rules clearly.

import express from 'express';
import { z } from 'zod';

const router = express.Router();
const createUserSchema = z.object({
  name: z.string().min(2),
  email: z.string().email()
});

router.post('/users', (req, res, next) => {
  try {
    const payload = createUserSchema.parse(req.body);
    res.status(201).json({ id: 1, ...payload });
  } catch (error) {
    next(error);
  }
});

router.use((error, req, res, next) => {
  res.status(400).json({ message: error.message });
});
  • Validate request body, params, and query values at the edge of the system.
  • Do not put business logic directly inside route definitions when the app grows.
  • Use one centralized error-shaping middleware instead of returning random formats.

Build a REST API with Clear Resource Design

A good Node.js tutorial should not stop at app.get('/'). Real APIs need resource naming, status codes, validation, service layers, and consistent responses.

const tasks = [
  { id: 1, title: 'Review invoice job', done: false }
];

app.get('/api/tasks', (req, res) => {
  res.json(tasks);
});

app.post('/api/tasks', (req, res) => {
  const task = {
    id: tasks.length + 1,
    title: req.body.title,
    done: false
  };

  tasks.push(task);
  res.status(201).json(task);
});

app.patch('/api/tasks/:id', (req, res) => {
  const task = tasks.find(item => item.id === Number(req.params.id));
  if (!task) return res.status(404).json({ message: 'Task not found' });

  task.done = Boolean(req.body.done);
  res.json(task);
});
REST rules worth keeping
  • Nouns for resources, not verbs for routes.
  • 201 for created resources, 404 when missing, 400 for invalid input.
  • Return one predictable response shape across the API.
  • Add pagination before large collections become painful.

GraphQL APIs with Node.js

GraphQL is useful when clients need flexible shapes of data, nested relationships, and fewer overfetching or underfetching problems than REST. It is powerful, but it needs discipline around schema design, validation, authorization, and query cost.

import { ApolloServer } from '@apollo/server';
import { startStandaloneServer } from '@apollo/server/standalone';

const tasks = [{ id: '1', title: 'Ship invoice export', done: false }];

const typeDefs = `#graphql
  type Task {
    id: ID!
    title: String!
    done: Boolean!
  }

  type Query {
    health: String!
    tasks: [Task!]!
  }
`;

const resolvers = {
  Query: {
    health: () => 'ok',
    tasks: () => tasks
  }
};

const server = new ApolloServer({ typeDefs, resolvers });
await startStandaloneServer(server, { listen: { port: 4000 } });
GraphQL rules that matter
  • Keep schema names business-focused, not database-focused.
  • Protect expensive nested queries with limits and auth checks.
  • Use DataLoader or batching to avoid N+1 database queries.
  • Log operation names and response time in production.

Practical choice: GraphQL is strongest when multiple clients need different slices of the same domain data. For a small CRUD service with simple endpoints, REST is often easier to operate.

Authentication: Sessions, JWT, Password Hashing, and Authorization

Auth in Node.js usually means four separate concerns: identity, password hashing, token or session management, authorization rules, and safe storage of secrets.

import bcrypt from 'bcryptjs';
import jwt from 'jsonwebtoken';

  return res.status(401).json({ message: 'Invalid credentials' });
}

const token = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '1h' }

res.json({ token });

Realtime Node.js with WebSockets and Socket.IO

Node.js is a natural fit for dashboards, chats, collaboration tools, live notifications, and background status updates because connections can stay open efficiently.

import express from 'express';
import http from 'node:http';
import { Server } from 'socket.io';

const app = express();
const server = http.createServer(app);
const io = new Server(server, { cors: { origin: '*' } });

io.on('connection', socket => {
  socket.on('chat:message', message => {
    io.emit('chat:message', {
      id: crypto.randomUUID(),
      text: message,
      createdAt: new Date().toISOString()
    });
  });
});

server.listen(3000);

Production note: realtime systems need sticky sessions or a shared adapter such as Redis when you scale to multiple instances.

Streams, Buffers, File Uploads, and Memory-Friendly Data Flow

Streams are how Node handles large files and chunked data efficiently. If you read a 2GB file fully into memory first, you are fighting the platform.

import { pipeline } from 'node:stream/promises';
import fs from 'node:fs';
import zlib from 'node:zlib';

await pipeline(
  fs.createReadStream('./report.csv'),
  zlib.createGzip(),
  fs.createWriteStream('./report.csv.gz')
);
Where streams matter
  • Large file downloads and uploads
  • CSV imports and exports
  • Compression and proxying
  • Image or media pipelines

Queues, Retries, Delayed Jobs, and Background Processing

Queues let your API return quickly while longer work happens in the background. This is where Node starts feeling production-ready: retries, backoff, concurrency limits, delayed jobs, and separate workers instead of brittle in-request side effects.

1Producer
import { Queue } from 'bullmq';

const emailQueue = new Queue('email-jobs', {
  connection: { host: '127.0.0.1', port: 6379 }
});

await emailQueue.add('send-receipt', {
  orderId: 'ORD-1042',
  email: 'customer@example.com'
}, {
  attempts: 5,
  backoff: { type: 'exponential', delay: 2000 }
});
2Worker
import { Worker } from 'bullmq';

const worker = new Worker('email-jobs', async job => {
  if (job.name === 'send-receipt') {
    await mailer.sendReceipt(job.data.orderId, job.data.email);
  }
}, {
  connection: { host: '127.0.0.1', port: 6379 },
  concurrency: 10
});
  • Use queues for emails, invoices, imports, thumbnails, webhooks, and third-party retry work.
  • Store enough job data to retry safely, but not so much that payloads become huge.
  • Make workers idempotent so a retry does not create duplicate side effects.

Reliability rule: never depend on in-memory arrays or delayed setTimeout() loops for business-critical background work. A real queue survives process restarts and supports retries intentionally.

Cron Jobs, Schedulers, and Repeatable Tasks

Cron-style jobs are useful for cleanup, digests, synchronization, billing windows, or health tasks. The tricky part is not syntax. It is making sure the job runs exactly where and when you expect in multi-instance production.

import cron from 'node-cron';

cron.schedule('0 * * * *', async () => {
  await syncInvoices();
  console.log('invoice sync complete');
}, {
  timezone: 'Asia/Kolkata'
});
Cron choices in production
  • Single instance app: in-process cron can be acceptable.
  • Multiple instances: use one scheduler service, a queue repeatable job, or external cron.
  • Critical finance or billing jobs: add locks, idempotency, and alerting.
  • Timezone-sensitive tasks: make the timezone explicit and test daylight-saving behavior.

Safe default: if duplicate execution would hurt users or money, do not trust a simple in-memory cron alone. Combine scheduling with distributed locks or a queue-based repeatable job.

Testing Node.js Apps with node:test, Supertest, and Practical Coverage

Modern Node includes a built-in test runner. That is enough for many services, especially when paired with Supertest for HTTP assertions. Use Vitest or Jest only when the extra ecosystem is worth it.

import test from 'node:test';
import assert from 'node:assert/strict';

test('sum adds numbers', () => {
  assert.equal(2 + 3, 5);
});
import request from 'supertest';
import { app } from '../src/app.js';

test('GET /api/health returns ok', async () => {
  const response = await request(app).get('/api/health');
  expect(response.statusCode).toBe(200);
  expect(response.body.ok).toBe(true);
});
  • Unit test pure business logic first.
  • Integration test API routes with a real request boundary.
  • Mock external services, not your own entire application.

Debugging, Logging, Environment Files, and Failure Analysis

Good Node debugging combines runtime flags, structured logs, meaningful error messages, and clear operational metadata. Random console.log() output everywhere is not a production strategy.

node --inspect src/server.js
node --inspect-brk src/server.js
node --trace-warnings src/server.js
node --env-file=.env src/server.js
import pino from 'pino';

const logger = pino({ level: process.env.LOG_LEVEL ?? 'info' });

logger.info({ requestId: req.id, route: req.path }, 'incoming request');
logger.error({ error }, 'payment capture failed');

Operational habit: log request IDs, user IDs when allowed, latency, route names, and external dependency failures. Otherwise production bugs become guessing games.

Node.js Security Basics for Real Apps

Security is mostly boring discipline: validate input, hash passwords, sanitize file paths, rate-limit public endpoints, use least-privilege secrets, and keep dependencies updated.

import helmet from 'helmet';
import cors from 'cors';
import rateLimit from 'express-rate-limit';

app.use(helmet());
app.use(cors({ origin: ['https://app.example.com'] }));
app.use(rateLimit({ windowMs: 15 * 60 * 1000, max: 200 }));
  • Keep secrets out of source control and out of client-visible bundles.
  • Validate every input boundary and reject unknown fields where possible.
  • Run npm audit, but do not blindly auto-upgrade without testing.
  • Use HTTPS, secure cookies, and trusted proxy settings in production.

Performance, Profiling, Caching, and Process Strategy

Performance work in Node usually means removing event-loop blocking, reducing duplicate I/O, using caching deliberately, and profiling before rewriting architecture.

node --cpu-prof src/server.js
node --heap-prof src/server.js
npx autocannon http://localhost:3000/api/health
Fast wins before major rewrites
  • Move expensive synchronous work off the request path.
  • Batch queries and remote calls with Promise.all() when safe.
  • Cache hot reads with Redis or in-memory layers where appropriate.
  • Use PM2 or orchestration to run multiple processes across CPU cores.

Profiling rule: use data before opinions. Node offers CPU and heap profiling flags so you do not have to guess where the time or memory went.

import { z } from 'zod';
import { ordersService } from '../services/orders.service.js';

const orderSchema = z.object({
  customerEmail: z.string().email(),
  items: z.array(z.object({ productId: z.number(), qty: z.number().min(1) }))
});

export async function createOrder(req, res, next) {
  try {
    const payload = orderSchema.parse(req.body);
    const order = await ordersService.create(payload);
    res.status(201).json(order);
  } catch (error) {
    next(error);
  }

// src/services/orders.service.js
import { prisma } from '../lib/prisma.js';
import { receiptQueue, inventoryQueue } from '../jobs/queues.js';

export const ordersService = {
  async create(payload) {
    const order = await prisma.order.create({
      data: {
        customerEmail: payload.customerEmail,
        status: 'pending',
        items: {
          create: payload.items.map(item => ({
            productId: item.productId,
            qty: item.qty
          }))
        }
      },
      include: { items: true }
    });

    await Promise.all([
      receiptQueue.add('send-receipt', { orderId: order.id, email: order.customerEmail }),
      inventoryQueue.add('reserve-stock', { orderId: order.id, items: payload.items })
    ]);

    return order;
  }
};

import { redis } from '../cache/redis.js';
import { prisma } from '../lib/prisma.js';

export async function getProduct(productId) {
  const key = `product:${productId}`;
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const product = await prisma.product.findUnique({ where: { id: productId } });
  await redis.setEx(key, 120, JSON.stringify(product));
  return product;

// common/redis/redis.service.ts
import { Injectable } from '@nestjs/common';
import { createClient } from 'redis';

@Injectable()
export class RedisService {
  client = createClient({ url: process.env.REDIS_URL });

  async onModuleInit() {
    await this.client.connect();
  }

  async getOrSet(key: string, ttl: number, loader: () => Promise<unknown>) {
    const cached = await this.client.get(key);
    if (cached) return JSON.parse(cached);
    const value = await loader();
    await this.client.setEx(key, ttl, JSON.stringify(value));
    return value;
  }
}
